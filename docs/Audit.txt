## 32. Instructions for the AI Building the Website: Static vs Dynamic CMS-Aware Architecture
When reviewing the repo and when proposing improvements, also define the exact implementation instructions that should be given to an AI engineer building the website so that static parts remain clean in code and dynamic parts are modeled in a way the CMS can recognize, manage, and inject safely.

Your job is not only to audit the current implementation, but to specify how the website should be built going forward.

### A. Core Rule: Separate Static Structure from Dynamic Content
The website must be built with a clear distinction between:

#### Static parts
These should usually live in code:
- global layout shell
- header structure
- footer structure
- navigation framework
- route structure
- page templates
- shared UI components
- spacing/layout system
- design system
- reusable rendering components
- access logic
- business logic
- API integrations

- localization infrastructure
- analytics/event hooks
- content rendering guards and fallbacks

#### Dynamic parts
These should be designed so the CMS can manage them:
- blog feeds and article listings
- popups and modal campaigns
- announcements and banners
- hero sections
- reusable marketing blocks
- callouts
- FAQ sections
- promotional strips
- SEO text blocks
- legal/update notices
- navigation/footer content values
- landing page sections
- localized content variants
- campaign content
- time-based content
- audience-targeted content
- page body sections
- reusable content slices
- media assets and file references

The AI building the website must not hardcode dynamic marketing/editorial content into templates unless there is a deliberate fallback.

### B. Instructions to Give the AI When Building Static Parts
When building static parts such as headers, footers, layouts, routes, and base pages, follow these rules:

1. Build static parts as structural components, not as content containers with hardcoded copy.
2. Keep the layout, styling, accessibility, and responsive behavior in code.
3. Replace editable text, links, badges, notices, CTA labels, nav items, and repeated marketing copy with CMS-bound fields where those values may change over time.
4. Build header and footer as schema-backed renderers:
   - structure can be coded
   - content items should be CMS-driven where business users may want to change them
5. Build pages using template/layout components that accept structured content models.
6. Do not bury page content inside JSX/HTML templates if editors are expected to update it.
7. Keep business rules and rendering logic separate from content data.
8. Use explicit component contracts for any section that may become CMS-driven later.
9. Create reusable page-section components that can render from structured CMS payloads.
10. Ensure every potentially editable section has a stable content key, type, or schema identifier.
11. Hardcode only what should truly remain stable:
   - app shell
   - technical routes
   - auth flows
   - dashboard mechanics
   - protected app logic
   - low-change legal framework structure
   - core product UI patterns

### C. Instructions to Give the AI When Building Dynamic Parts
When building dynamic parts, instruct the AI to make them CMS-recognizable and CMS-manageable by design.

For every dynamic element:
- define a content type
- define fields explicitly

- define rendering component mapping
- define fallback behavior
- define localization behavior where needed

### D. CMS Recognition Rules
Tell the AI to build the system so the CMS can recognize editable content reliably.

That means:
1. Every dynamic section must have a content type or block type.
2. Every block must have a unique type identifier.
3. Every page that supports CMS injection must define editable regions or slots.
4. Templates must declare which block types are allowed in which regions.
