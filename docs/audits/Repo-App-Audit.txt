# platform Repo + App Audit

Date: 2026-04-18

## Executive Summary

Current status: not release-ready.

The backend is close to healthy, but the project still has release-blocking issues in three places:

1. The frontend production build is broken after a clean `npm install`.
2. The dashboard shell assumes optional modules/tables exist and can 500 when they do not.
3. The editor fallback preview regressed for TSX/JSX source pages, which breaks a key recovery path for unbuilt runtime projects.

The good news is that the backend test suite is mostly green once run outside the sandbox: `181 passed`, `3 failed`, `1 skipped`.

## Findings

### 1. Release blocker: frontend production build is broken after clean install

Severity: Critical

Evidence:

- `npm install` completed successfully.
- `npm.cmd run build` still failed.
- Failure chain points into `@rolldown/pluginutils` importing `../utils.js`, but that file is not present in the installed package.

Relevant files:

- `package.json`
- `package-lock.json` around the `vite`, `rolldown`, and `@vitejs/plugin-react` entries
- `node_modules/@rolldown/pluginutils/dist/filter/composable-filters.js`

Why this matters:

- A clean workspace still cannot produce a production frontend bundle.
- That makes deployments and CI builds unreliable or impossible.

Likely root cause:

- The current dependency set resolves to `vite@8.0.8`, `rolldown@1.0.0-rc.15`, and `@vitejs/plugin-react@6.0.1`.
- The resolved Rolldown utility package on disk is internally inconsistent.

Recommendation:

- Pin to a known-good Vite/plugin-react stack, or downgrade from the broken Rolldown RC chain.
- Re-run `npm install` and `npm.cmd run build` after the version fix.

### 2. Dashboard shell can crash when optional tables are missing

Severity: High

Evidence:

- `Tests/Feature/DashboardSchemaCompatibilityTest` fails with:
  `SQLSTATE[HY000]: General error: 1 no such table: site_inbox_messages`
- The failing query is built in the Inertia shared-props middleware, not only in the dashboard route.

Root cause:

- [app/Http/Middleware/HandleInertiaRequests.php](<workspace>\app\Http\Middleware\HandleInertiaRequests.php:24) unconditionally adds `withCount()` clauses for:
  - `inboxMessages as unread_inbox_count`
  - `invoices as unpaid_invoices_count`
  - `reminders as overdue_reminders_count`

Why this matters:

- Any Inertia page that depends on shared `navSites` can fail if one of those optional tables is absent during partial installs, schema drift, or compatibility scenarios.
- This is broader than a single dashboard page problem.

Recommendation:

- Guard these counts with schema checks before building the query.
- Keep the shared Inertia nav logic aligned with the guarded Blade composer logic in [app/Providers/AppServiceProvider.php](<workspace>\app\Providers\AppServiceProvider.php:49).

### 3. Source fallback preview is broken for TSX/JSX pages

Severity: High

Evidence:

- `Tests/Feature/EditorPreviewFallbackRegexTest` fails.
- Expected fallback preview content does not render; the controller returns the generic "Preview unavailable" page instead.

Root cause:

- [app/Http/Controllers/EditorPreviewController.php](<workspace>\app\Http\Controllers\EditorPreviewController.php:455) strips JSX expressions with:
  - `preg_replace('/\{[^{}]*\}/', ' ', $source)`
  - repeated twice
- For a typical TSX component, that can remove the entire function body before snippet extraction runs, leaving no fallback content.

Why this matters:

- Visual editing for runtime-framework pages depends on this source fallback when built output is unavailable.
- Today that recovery path silently degrades into a dead-end state for valid TSX pages.

Recommendation:

- Narrow the JSX expression stripping so it does not erase whole component bodies.
- Add a regression case for real component bodies, not only symbol handling.

### 4. Static analysis is not trustworthy right now

Severity: Medium

Evidence:

- `composer phpstan` fails with `18` errors.
- Some are real, some are stale ignores that no longer match the code.

Concrete issues:

- [app/Providers/FortifyServiceProvider.php](<workspace>\app\Providers\FortifyServiceProvider.php:14) imports `UpdateUserPasswordResponse` and `UpdateUserProfileInformationResponse`, but the installed Fortify version exposes `PasswordUpdateResponse` and `ProfileInformationUpdatedResponse` instead.
- [app/Livewire/Dashboard/SiteHealthTable.php](<workspace>\app\Livewire\Dashboard\SiteHealthTable.php:43) still has a real collection callback typing issue.
- [phpstan-baseline.neon](<workspace>\phpstan-baseline.neon:175) contains multiple ignore patterns for `SiteHealthTable` and `FileManager` that no longer match current errors.

Why this matters:

- CI static analysis noise makes real regressions easier to miss.
- The current baseline is masking drift instead of documenting it.

Recommendation:

- Fix the Fortify response contract names to match the installed package.
- Resolve the `SiteHealthTable` map typing issue.
- Regenerate or trim the stale baseline entries.

### 5. One failing feature test is stale rather than proving an app bug

Severity: Low

Evidence:

- [tests/Feature/SiteShowDashboardTest.php](<workspace>\tests\Feature\SiteShowDashboardTest.php:101) uses `assertSee('Visitors today')` and similar HTML assertions.
- The site dashboard labels do exist in [resources/js/pages/sites/show.tsx](<workspace>\resources\js\pages\sites\show.tsx:112).
- The root view is a standard Inertia mount in [resources/views/app.blade.php](<workspace>\resources\views\app.blade.php:10), so those labels are rendered client-side, not in the initial response HTML.

Why this matters:

- This keeps the suite red even when the page contract is likely correct.
- It obscures the higher-value failures above.

Recommendation:

- Rewrite this test to use Inertia assertions against component name and props instead of raw HTML text checks.

## Secondary Risks

### Duplicate nav-site query logic

There are two separate implementations for nav-site counts:

- [app/Http/Middleware/HandleInertiaRequests.php](<workspace>\app\Http\Middleware\HandleInertiaRequests.php:18)
- [app/Providers/AppServiceProvider.php](<workspace>\app\Providers\AppServiceProvider.php:49)

This duplication has already drifted. The Blade path is schema-guarded; the Inertia path is not.

### Schema helper cache can go stale inside long-lived processes

[app/Support/SchemaState.php](<workspace>\app\Support\SchemaState.php:12) caches table/column existence statically for the lifetime of the PHP process.

That is fast, but risky during compatibility tests, migrations, or long-lived workers if schema changes without a full process restart.
