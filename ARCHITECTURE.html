Why this stack? Laravel + Livewire keeps everything in PHP — you write reactive dashboard components without touching a JS framework. Section 02

System Architecture

pixelkraft is a monolithic Laravel application with heavy use of background job processing. The system has three main layers:

Dashboard Layer

Livewire + Alpine.js

Server-rendered reactive UI. Site listing, page editor, SEO tools, analytics panels, file manager, settings. Communicates with backend via Livewire's wire protocol (no REST API needed for internal use).

Application Layer

Laravel Services

Business logic organized as domain services: GitSyncService, ParserService, DeployService, SeoAnalyzer, AnalyticsAggregator, etc. Each service is independently testable and injected via Laravel's container.

Worker Layer

Horizon + Queue Jobs

All heavy processing runs asynchronously: git clone/pull/push, project builds, Lighthouse audits, uptime checks, image optimization, sitemap generation, broken link crawling, backups, email sending.

Data Layer

MariaDB + Redis + R2

MariaDB stores all structured data (sites, pages, regions, content, analytics, logs). Redis handles queues, caching, and sessions. Cloudflare R2 stores all media assets with per-site isolation.

Section 04

Multi-Strategy Parser

This is the hardest engineering challenge in the entire project. Your sites span plain HTML, React, Vue, Svelte, Astro, Hugo, and 11ty. A single parsing approach will not work. The parser must detect the project type and apply the right strategy.

Step 1: Project Type Detection

When a repo is cloned, the ProjectDetector service scans for signature files:

// Detection rules (checked in order) package.json contains "astro" → astro package.json contains "next" → react (Next.js) package.json contains "nuxt" → vue (Nuxt) package.json contains "svelte" → svelte package.json contains "react" → react package.json contains "vue" → vue hugo.toml or config.toml exists → hugo .eleventy.js or eleventy.config.js exists → 11ty *.html files in root → static_html fallback → custom (manual config required)

Step 2: Parse Strategy per Type

Easy

Static HTML

Parse directly with Symfony DomCrawler. Every .html file = one page. Full DOM access. This is the golden path — most accurate parsing possible.

Medium

SSG Output (Hugo, Astro, 11ty)

Run the build command → parse the output HTML (same as static). But edits must map back to source files (markdown, frontmatter, template partials). Requires a source-to-output map.

Hard

SPAs (React, Vue, Svelte)

Source HTML is nearly empty. Content lives in .jsx/.vue/.svelte component files. Parse component source to find text/props. Use Browsershot to render the actual page for the visual editor.

Universal

Browsershot Fallback

For any project type: run the build, serve it locally, render with headless Chromium. Captures the real DOM as the user would see it. Used for visual editor rendering and screenshots.

Step 3: Region Detection Heuristics

For each parsed page, the RegionDetector scores every element:

// Scoring factors (sum → confidence score 0-1) +0.3 Semantic content tag (h1, h2, h3, p, article, section, figcaption) +0.2 Contains meaningful text (>20 chars, not just whitespace) +0.2 Inside a <main> or [role="main"] container +0.1 Is an <img> with src and alt attributes +0.1 Has a unique, descriptive class/id (not generic like "container") +0.1 Is a <a> with visible text content -0.3 Inside <nav>, <header>, or <footer> (likely static/theme) -0.2 Contains mostly other elements (structural wrapper) -0.2 Repeats across multiple pages (nav menu, footer) -0.1 Has inline event handlers (likely interactive/functional) -0.1 Is a <script> or <style> tag // Score >= 0.5 → auto-classified as dynamic (editable) // Score < 0.5 → auto-classified as static (locked) // All classifications presented in dashboard for user confirmation

Step 4: Marker Injection

After user confirms regions in the dashboard, markers are injected into source files:

<!-- Before --> <h1 class="hero-title">Welcome to My Site</h1> <!-- After confirmation --> <!-- cms:editable id="hero-title" type="text" --> <h1 class="hero-title">Welcome to My Site</h1> <!-- /cms:editable --> // Subsequent parses use markers first (instant), then auto-detect for new elements
Section 05

Visual Page Editor

The editor is the core user experience. It renders the actual website in an iframe with an interactive overlay that lets you click on elements, see their type (static/dynamic), and edit them.

Architecture

Parent Frame

Editor Chrome (Livewire + Alpine)

Sidebar with page info, region list, editor panels. Toolbar with save/preview/code toggle. Communicates with iframe via postMessage. Alpine.js handles local UI state.

Child Frame

Rendered Page + Injected Script

The actual website loaded in an iframe. A small JS script is injected that intercepts clicks, highlights hovered elements, shows region boundaries, and sends selected element data back to parent.

Editing Modes

  • Visual Mode: Click an element in the rendered page. If it's a dynamic region, an inline editor appears (rich text for text nodes, image picker for images, link editor for anchors). Changes are reflected live in the iframe.
  • Code Mode: Full file source code displayed in a code editor (CodeMirror via Livewire). Syntax highlighting for HTML/CSS/JS. The selected region is highlighted in the source. Edits here update the visual preview.
  • Static regions: Clicking a static region shows a lock icon and grayed-out overlay. A "Convert to editable" button lets you override classification.
  • Toggle: A button in the toolbar switches between visual and code view. Current cursor/selection position is preserved across modes where possible.

Section 06

GitHub Sync Engine

The sync engine manages local clones of all site repos on the VPS. It uses CzProject/GitPHP for local git operations and GitHub's webhook API for remote event listening.

Operations

  • Clone: When a site is added, clone the repo to /var/www/pixelkraft/repos/{site-slug}/. Dispatched as a queue job.
  • Pull: Before any edit session begins, pull latest changes. Also triggered by webhook when someone pushes directly to GitHub.
  • Commit: After save, stage changed files, create commit with auto-generated message (e.g., "Update hero title on /about"). User can customize message.
  • Push: Push commit to origin. If push fails (remote has new commits), auto-pull and rebase, then retry. If conflict, surface it in the dashboard.
  • Webhook listener: A dedicated route receives GitHub push events. Validates signature, pulls changes, re-parses affected files, updates regions.

Conflict handling: Since you're the sole user, conflicts are rare but possible (e.g., editing in CMS while also pushing from your local machine). The CMS will detect conflicts, show a diff view, and let you choose which version to keep.

Deployment Pipeline

pixelkraft replaces GitHub Actions entirely. Each site has a configurable pipeline:

Fetch latest from origin

02

03
Build

Run configured build command

04

05
Deploy

Copy to serve directory, reload Nginx

Per-Site Configuration

// Stored in sites table, configured in dashboard build_command: "npm run build" // or "hugo", "npx astro build", null for static build_output_dir: "dist" // or "public", "build", "_site", "." for static node_version: "20" // managed via nvm env_variables: {"API_URL": "..."} // injected during build deploy_path: "/var/www/sites/mysite" // Nginx serves from here pre_deploy_hook: null // optional shell command post_deploy_hook: null // optional shell command

Staging & Rollback

  • Staging: Before deploying, build output is served on a temporary subdomain (staging-{slug}.cms.yourdomain.com) for preview. You approve or reject.
  • Rollback: Each successful deploy is tagged with a timestamp. Rollback = git checkout that tag, re-deploy. Keeps last 10 deploy snapshots per site.
Section 08

Domain & SSL Management

  • Add domain: Enter domain in dashboard → CMS generates Nginx vhost config → writes to /etc/nginx/sites-available/ → symlinks to sites-enabled/ → reloads Nginx.
  • SSL provisioning: After Nginx is configured, CMS runs certbot --nginx -d domain.com via a queue job. Stores cert expiry in database. Renews automatically via cron (certbot handles this natively).
  • SSL monitoring: Weekly scheduled job checks all cert expiry dates. Alerts you via Discord + in-app notification 14 days before expiry.
  • Nginx templates: CMS uses Blade templates to generate vhost configs. Different templates for static sites vs. proxied apps. Includes redirect rules from the 301 manager.

Security note: The Laravel process needs carefully scoped sudo permissions to write Nginx configs, run certbot, and reload Nginx. This is handled via a sudoers entry that only allows specific commands, not blanket root access.

Section 09

Content Management

Two content models coexist in pixelkraft:

Region-Based

Editable Regions

For existing pages: click on elements, edit in place. Good for landing pages, about pages, static content. Changes are patched directly into source HTML/template files.

Structured

Collection Content

For blogs and products: fill in fields (title, body, image, tags, price). CMS generates the HTML file from a template and writes it to the correct location in the repo.

Blog Post Workflow

  • Click "New Post" on a site → structured editor opens with fields: title, slug, body (rich text), excerpt, featured image, tags, SEO fields
  • Body editor is TipTap-based (Livewire component) for rich text, with toggle to raw HTML/markdown
  • On save: CMS renders the post through the site's blog template → writes the output HTML to the configured blog directory in the repo → commits and deploys
  • Existing hard-coded blog posts can be imported: parser detects repeated page structures (blog listing patterns) and offers to convert them to managed posts

Content Templates

  • Reusable templates with placeholder fields: {{title}}, {{body}}, {{image}}
  • Templates can be per-site or global (shared across sites)
  • Types: page templates, section templates (reusable blocks), component templates (header/footer)
  • Global components: define a shared header/footer as a template, and CMS injects it across all pages of a site during build
Section 10

SEO Suite

Every page in the dashboard shows an SEO panel with all metadata editable in one place:

  • Meta editor: Title tag, meta description, keywords. Character count with recommended limits. Preview of how it looks in Google search results.
  • Open Graph: og:title, og:description, og:image, og:type. Preview card for Facebook/Twitter/LinkedIn sharing.
  • JSON-LD: Schema.org structured data editor. Preset templates for Article, Product, LocalBusiness, FAQ, etc. Raw JSON editor for custom schemas.
  • robots.txt: Per-site editor with common presets (allow all, block AI bots, etc.).
  • Canonical URLs: Set per page. Auto-generated for blog posts based on slug.
  • 301 Redirects: Add/edit/delete redirect rules. Stored in database, injected into Nginx vhost config on deploy.
  • Sitemap: Auto-generated XML sitemap per site. Updated on every deploy. Submitted to Search Console if connected.
  • SEO Score: Computed 0-100 score per page based on: title length, description presence, heading hierarchy, image alt tags, canonical URL, schema markup presence.
Section 11

Analytics & Monitoring

Unified Analytics Dashboard

Pulls data from Google Analytics (via GA4 Data API) and Cloudflare Analytics (via GraphQL API) into a single view. Data is synced daily via scheduled jobs and cached in analytics_snapshots table.

  • Site-level metrics: Total visitors, pageviews, bounce rate, top pages, traffic sources — combining GA + Cloudflare data
  • Page-level metrics: Each page in the page listing shows its visit count, visible in the list columns
  • Custom event tracking: Define custom events per site (button clicks, form submissions, scroll depth). Tracked via GA and displayed in dashboard
  • Google Search Console: Keyword rankings, click-through rates, indexing status, crawl errors. Synced via Search Console API

Monitoring

  • Uptime monitoring: HTTP check every 5 minutes per site. Stores response code + response time. Alerts on 3 consecutive failures via Discord + in-app notification
  • Lighthouse audits: Weekly scheduled Browsershot-powered Lighthouse runs per page. Stores performance, accessibility, best practices, SEO scores. Trend graphs in dashboard
  • Broken link checker: Weekly crawl of all pages per site. Reports 404s, redirect chains, and broken external links. Results shown in an "Issues" panel per site
Section 12

Email System

Contact Form Submissions

  • API endpoint: POST https://cms.yourdomain.com/api/forms/{site-slug} — accepts JSON body with any fields. Rate-limited, CORS-configured per site, honeypot spam detection.
  • Dashboard view: Per-site inbox showing all submissions. Mark as read, star, delete. Filter by form name, date, spam status.
  • Notifications: New submission → Discord notification with form data summary.

Newsletter System

  • Subscriber management: Per-site subscriber lists. Import/export CSV. Manual add. Segments/tags for targeting.
  • Campaign editor: Rich text email composer built with Livewire. Pre-built templates (announcement, update, promotion). Subject line, preview text, body.
  • Scheduling: Send immediately or schedule for a future date/time. Queue processes via Horizon.
  • Basic segmentation: Target by tag, signup date, or custom segment.
  • Sending: Via Resend API. Laravel's built-in Mail system with Resend driver. Handles bounces and unsubscribes via Resend webhooks.
  • Stats: Sent, delivered, opened, clicked, bounced — per campaign. Displayed in dashboard.
Section 13

Media Management

  • Per-site media library: Each site has isolated storage in Cloudflare R2 under prefix sites/{site-slug}/media/
  • Upload: Drag-and-drop or file picker in the editor and media library panel. Files uploaded to R2 via Laravel's Filesystem (S3 driver works with R2)
  • Auto-optimization on upload: Images are processed through the optimization pipeline (see Performance section) before being stored in R2
  • Browsing: Grid view of all media for a site with filename, dimensions, file size, upload date. Search and filter.
  • Usage tracking: Track which pages reference each media file. Warn before deleting a file that's in use.
  • Image picker in editor: When editing an image region, a modal opens showing the site's media library with upload option.
Section 14

Performance Optimization

Runs as a step in the deployment pipeline, after build but before files are copied to the serve directory.

  • Image optimization: Spatie Image Optimizer (wraps jpegoptim, pngquant, svgo). Runs on all images in build output. Also generates WebP variants using cwebp. Original preserved, WebP served via Nginx try_files with Accept header detection.
  • Lazy loading: DomCrawler scans built HTML, adds loading="lazy" to all <img> tags below the fold (skip first 2 images per page as they're likely above-fold). Adds decoding="async" as well.
  • Minification: HTML minification via voku/html-min. CSS via tubalmartin/cssmin. JS via tedivm/jshrink. All are PHP-native, no Node dependency for this step.
  • These optimizations are applied to the deploy output only — source files in the repo remain unmodified and readable.
Section 15

Operations & Logging

Deploy Logs

Every deploy is logged with full output (git pull output, build stdout/stderr, optimization results, deploy result). Viewable per-site in the dashboard with timestamp, duration, status, commit SHA, and trigger type (manual, save, webhook).

Error Logs

Per-site Nginx error logs are tailed and stored. Laravel's own logs are accessible. Each site has an "Issues" tab showing recent errors, broken links, and failed jobs. All errors are aggregated in a global "Issues" panel on the home dashboard.

Automated Backups

  • Database: Daily MariaDB dump via spatie/laravel-backup. Stored in Cloudflare R2 under backups/db/. Retains last 30 days.
  • Media: R2 handles durability natively (11 nines). No additional backup needed for media files.
  • CMS config: The entire CMS application is in a GitHub repo itself, so code is backed up. Database + media covers everything else.

Notifications

  • In-dashboard: Bell icon with unread count. Notification types: deploy_success, deploy_failed, ssl_expiring, uptime_down, form_received, lighthouse_drop, broken_links_found
  • Discord: Configurable webhook URL per user. Sends formatted embeds for critical alerts (failures, downtime, SSL). Uses Laravel's notification system with a custom Discord channel.

Section 17

Auth & Security

  • Authentication: Email/password with bcrypt hashing. Built on Laravel Fortify.
  • 2FA: TOTP-based (Google Authenticator, Authy, etc.) via Laravel Fortify's built-in 2FA support. Recovery codes generated on setup.
  • Future team support: Users table has a role column (admin/editor). Admin = full access. Editor = can edit content but not manage domains, SSL, or server config. Role-based middleware ready from day one.
  • API tokens: Laravel Sanctum for API authentication. Tokens created/revoked in settings.
  • Session security: Sessions stored in Redis. CSRF protection on all forms (Livewire handles this). Rate limiting on login (5 attempts/minute).
  • GitHub tokens: Encrypted at rest using Laravel's encryption (AES-256-CBC).
