Scope
- Deep-dive audit of the Laravel app with emphasis on real breakages, security issues, and operational hazards.
- I used source inspection plus targeted runtime/test reproductions.
- I did not change application code for this audit.

Baseline
- `php artisan test --compact` on 2026-04-15 finished with 16 failed and 165 passed tests.
- Several failures are caused by real product defects.
- Several others are stale or broken tests that currently hide the product signal.

Status tags used below
- REPRODUCED: I confirmed the behavior by running code or tests.
- CODE AUDIT: high-confidence finding from source inspection.
- TEST DEBT: current test failure that does not appear to reflect a shipped app defect.

1. [P1] [REPRODUCED] Tracking ingestion crashes on normal event payloads
Impact
- Frontend analytics collection is effectively broken for normal pageview and interaction events.
- Requests return HTTP 500 before the event is recorded.

Evidence
- `app/Http/Controllers/TrackingController.php:27-35` validates `payload.*` with `scalar`.
- Laravel's validator does not ship a built-in `scalar` validation rule.
- `php artisan test --compact` reproduced this in `Tests\Feature\TrackingControllerTest` with:
  `BadMethodCallException: Method Illuminate\Validation\Validator::validateScalar does not exist.`

Relevant refs
- `app/Http/Controllers/TrackingController.php:27-35`

Recommendation
- Replace `scalar` with supported validation that matches the intended payload contract.
- Add a feature test that posts the same metadata shape the shipped tracker emits.

2. [P1] [REPRODUCED] Visual editor code mode can write files outside the repository
Impact
- An authenticated user who can reach the visual editor's code mode can escape the site repo and write arbitrary files elsewhere on disk.
- This is a filesystem write primitive, not just a validation bug.

Evidence
- `app/Livewire/Editor/VisualEditor.php:371-374` writes `File::put($fullPath, $this->codeContent)`.
- `app/Livewire/Editor/VisualEditor.php:604-634` tries to guard with `realpath($candidate) ?: realpath(dirname($candidate))`.
- If the target file and parent directory do not exist yet, the guard falls back to the repo root and returns the original traversal path unchanged.

Reproduction
- I bootstrapped the app, instantiated `VisualEditor`, set:
  `codeFilePath = '..\\..\\outside\\newdir\\pwned.txt'`
- `resolveCodePath()` returned:
  `C:\Users\agonv\AppData\Local\Temp\pk_audit_...\repo/..\..\outside\newdir\pwned.txt`
- `File::put()` succeeded.
- A verification check showed `INSIDE_REPO=no`.

Relevant refs
- `app/Livewire/Editor/VisualEditor.php:371-374`
- `app/Livewire/Editor/VisualEditor.php:604-634`

Recommendation
- Canonicalize the final intended path before writing.
- Reject any path containing traversal segments before filesystem creation.
- Add a regression test for a non-existent escaped parent path, not just existing-path traversal.

3. [P1] [REPRODUCED] Runtime deployments can come up on one port while Nginx proxies another
Impact
- Runtime sites can become unreachable after deploy when the preferred port is already occupied.
- This is an outage path for Next.js runtime deployments.

Evidence
- `app/Services/SiteRuntimeService.php:149-158` deploys on `$port = $this->allocatePort($site)`.
- `app/Services/NginxConfigService.php:227-253` proxies to `$this->runtime->portFor($site)`.
- Those are not the same thing when the preferred port is busy.

Reproduction
- I created a temporary listener on the site's preferred runtime port.
- `SiteRuntimeService::allocatePort()` returned the next free port.
- `NginxConfigService::renderRuntimeTemplate()` still rendered `proxy_pass` using the original preferred port.
- Result of the reproduction script:
  `PREFERRED=4929`
  `ALLOCATED=4930`
  `NGINX_PROXY_PORT=4929`
  `MISMATCH=yes`

Secondary risk in the same area
- `app/Services/SiteRuntimeService.php:283-295` runs standalone apps as `node server.js` with port passed in the environment.
- `app/Services/SiteRuntimeService.php:549-553` only rediscovers ports from `PORT=` or `-p <port>` in the command line.
- That means active-port rediscovery can miss standalone processes after restart or cache loss.

Relevant refs
- `app/Services/SiteRuntimeService.php:149-158`
- `app/Services/SiteRuntimeService.php:283-295`
- `app/Services/SiteRuntimeService.php:549-553`
- `app/Services/NginxConfigService.php:227-253`

Recommendation
- Persist and use the actual allocated runtime port everywhere, including generated Nginx config.
- Add an end-to-end regression test covering a busy preferred port.

4. [P1] [CODE AUDIT] Site maintenance mode appears to fail open
Impact
- Operators can believe a site is protected by maintenance mode while the live site remains publicly reachable.
- The advertised IP allowlist bypass also appears unenforced.

Evidence
- `resources/views/livewire/sites/maintenance-mode.blade.php:4-12` describes a visitor-facing maintenance page.
- `app/Livewire/Sites/MaintenanceMode.php:75-89` persists `enabled`, `allowedIPs`, branding, and messaging.
- `routes/web.php:165-168` exposes a dashboard preview route.
- Repo-wide inspection did not find any runtime middleware, deploy step, or Nginx generation path that serves those settings to public traffic.
- The only concrete serving path I found was the preview page.

Important clarification
- `app/Console/Commands/AppDeploy.php` uses Laravel's global app maintenance mode.
- That is separate from the per-site maintenance UI above and does not implement the saved site-level settings.

Relevant refs
- `resources/views/livewire/sites/maintenance-mode.blade.php:4-12`
- `app/Livewire/Sites/MaintenanceMode.php:75-89`
- `routes/web.php:165-168`
- `app/Console/Commands/AppDeploy.php:19-27`

Recommendation
- Either wire the feature into live request serving and deployment, or relabel it as preview-only until enforcement exists.
- Add a feature/integration test that proves a live site returns the maintenance page when enabled.

5. [P2] [REPRODUCED] Login page crashes when registration is disabled
Impact
- The login page returns HTTP 500 in the default configuration.
- This blocks sign-in entirely in setups where open registration is intentionally off.

Evidence
- `config/fortify.php:146-149` disables registration by default unless `REGISTRATION_ENABLED=true`.
- `resources/views/auth/login.blade.php:31-32` always renders `route('register')`.
- `php artisan test tests/Feature/AuthPagesTest.php --stop-on-failure` reproduced:
  `Route [register] not defined.`

Relevant refs
- `config/fortify.php:146-149`
- `resources/views/auth/login.blade.php:31-32`

Recommendation
- Guard the register link behind `Route::has('register')` or the Fortify feature flag.
- Add a test for the login page with registration disabled.

Impact

- `resources/views/components/layouts/editor-shell.blade.php:21`

  `Vite manifest not found at: ...\public\build\manifest.json`

Observed affected tests
- `Tests\Feature\BlogEditRouteTest`
- `Tests\Feature\DashboardIndexTenancyTest`
- `Tests\Feature\SiteShowDashboardTest`
- `Tests\Feature\ExampleTest`

Relevant refs
- `resources/views/components/layouts/auth.blade.php:11`
- `resources/views/components/layouts/app.blade.php:11`
- `resources/views/components/layouts/editor-shell.blade.php:21`

Recommendation
- Decide whether tests should build assets, fake Vite, or use a fallback strategy in non-built environments.
- Add a documented local/CI bootstrap path so the app does not 500 on first run.

7. [P2] [CODE AUDIT] API tokens are minted as full-access, non-expiring credentials
Impact
- The settings UI creates broad long-lived API keys rather than scoped tokens.
- Any token issued here can hit deploy/sync/rollback routes that rely on Sanctum abilities.

Evidence
- `app/Livewire/Settings/ApiTokens.php:14-21` calls `createToken($this->tokenName)` with no abilities or expiry.
- `vendor/laravel/sanctum/src/HasApiTokens.php:60` defaults that to `['*']` and no expiration.
- `app/Http/Middleware/EnsureSanctumApiTokenCan.php` expects route-specific abilities.

Relevant refs
- `app/Livewire/Settings/ApiTokens.php:14-21`
- `vendor/laravel/sanctum/src/HasApiTokens.php:60`
- `app/Http/Middleware/EnsureSanctumApiTokenCan.php:54-67`

Recommendation
- Issue least-privilege tokens with explicit abilities and expiry.
- Expose token purpose/scope in the UI instead of a generic free-form name only.

8. [P2] [CODE AUDIT] System diagnostics route is exposed to any authenticated user
Impact
- Any authenticated user can reach internal queue, Redis, failure, and stuck-site diagnostics.
- This is an authorization gap and information-disclosure issue.

Evidence
- `routes/web.php:205-206` registers `/system` inside the authenticated dashboard group with no admin gate.
- `app/Livewire/Settings/SystemDiagnostics.php:23-76` assembles queue driver, Redis status, missing queues, pending jobs, failed jobs, and stuck-site summaries.
- Horizon itself is admin-gated elsewhere, so this route is looser than the rest of the ops surface.

Relevant refs
- `routes/web.php:205-206`
- `app/Livewire/Settings/SystemDiagnostics.php:23-76`
- `app/Providers/HorizonServiceProvider.php`

Recommendation
- Restrict the route to administrators or an ops capability.
- Review other dashboard settings routes for the same pattern.

9. [P3] [CODE AUDIT] CSP and layout font loading disagree
Impact
- The app references external font stylesheets that the CSP disallows.
- In browsers that enforce the current policy, those stylesheets will be blocked.

Evidence
- Layouts load:
  `https://fonts.bunny.net/...`
  `https://fonts.googleapis.com/...`
- `app/Http/Middleware/SetSecurityHeaders.php:72-78` sets:
  `style-src 'self' 'unsafe-inline'`
- That policy does not allow external stylesheet origins.

Relevant refs
- `resources/views/components/layouts/app.blade.php:8-10`
- `resources/views/components/layouts/auth.blade.php:8-10`
- `resources/views/components/layouts/editor-shell.blade.php:18-20`
- `app/Http/Middleware/SetSecurityHeaders.php:72-78`

Recommendation
- Either self-host those stylesheets/fonts or add the approved stylesheet origins to CSP intentionally.

10. [P3] [CODE AUDIT] Blog publisher has the same non-existent-parent path guard weakness pattern
Impact
- If a malicious or corrupted `output_path` is ever persisted, blog export can write outside the repo when the escaped parent directory does not yet exist.
- Current UI validation reduces reachability, but the service guard is still weaker than it should be.

Evidence
- `app/Services/BlogPostPublisher.php:82-99` builds `$fullPath = "{$repoPath}/{$outputPath}"`.
- If `realpath(dirname($fullPath))` fails, it falls back to the repo root and then calls `File::ensureDirectoryExists(dirname($fullPath))`.
- That is the same fail-open pattern confirmed in `VisualEditor`.

Relevant refs
- `app/Services/BlogPostPublisher.php:82-99`

Recommendation
- Harden the service-level path validation even if UI validation already blocks traversal strings.
